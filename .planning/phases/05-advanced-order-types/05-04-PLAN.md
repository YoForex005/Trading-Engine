<task type="auto">
  <name>Task 1: Implement OCO order linking and cancellation</name>
  <files>backend/internal/core/engine.go, backend/internal/api/handlers/orders.go</files>
  <action>
    In engine.go:
    - Add CancelOCOLinkedOrder(orderID int64) method
    - Load order via orderRepo.GetByID(orderID)
    - If oco_link_id is set:
      * Load linked order via orderRepo.GetByID(oco_link_id)
      * If linked order Status=PENDING:
        - Update linked order Status=CANCELLED
        - Set RejectReason="OCO order filled"
        - Save via orderRepo.Update()
        - Broadcast order update via WebSocket
    - Call from ExecutePendingOrder() after position created (Plan 03)
    - Call from ExecuteTriggeredOrder() after position closed (Plan 01 SL/TP)

    In handlers/orders.go:
    - Update POST /api/orders to accept ocoLinkId parameter
    - When creating order with ocoLinkId:
      * Validate linked order exists and is PENDING
      * Set both orders' oco_link_id to point to each other:
        - New order: oco_link_id = ocoLinkId
        - Linked order: update oco_link_id = new_order.ID
      * Save both via orderRepo
    - Return both order IDs in response

    Use case example (bracket order):
    1. Place Buy Stop at 1.10 with ocoLinkId=null → Order A (ID=1)
    2. Place Buy Limit at 1.08 with ocoLinkId=1 → Order B (ID=2), both link to each other
    3. Price rises to 1.10 → Order A fills, Order B cancelled automatically
  </action>
  <verify>
    - go build ./backend/... succeeds
    - Manual test: Create two linked pending orders → fill one → other cancels
    - Check database: cancelled order has CANCELLED status and OCO reject reason
  </verify>
  <done>
    - OCO linking creates bidirectional relationship
    - Filling one order cancels the other automatically
    - Cancelled order has correct status and reason
    - Changes persist to database
  </done>

<task type="auto">
  <name>Task 2: Implement order modification and expiration</name>
  <files>backend/internal/api/handlers/orders.go, backend/internal/core/order_monitor.go</files>
  <action>
    In handlers/orders.go:
    - Add PATCH /api/orders/:id endpoint
    - Parse request body:
      {
        "triggerPrice": float64 (optional),
        "sl": float64 (optional),
        "tp": float64 (optional),
        "volume": float64 (optional),
        "expiryTime": timestamp (optional)
      }
    - Validation:
      * Order exists and Status=PENDING
      * If triggerPrice changed: validate still makes sense for order type
      * Volume > 0 if changed
    - Update order fields and save via orderRepo.Update()
    - Return updated order

    - Add DELETE /api/orders/:id endpoint
    - Validate order is PENDING
    - Update Status=CANCELLED, RejectReason="User cancelled"
    - Save via orderRepo.Update()
    - Return success

    In order_monitor.go:
    - Add CheckOrderExpiry() method
    - For each order with expiry_time set:
      * If time.Now() > expiry_time and Status=PENDING:
        - Update Status=CANCELLED, RejectReason="Expired"
        - Save via orderRepo.Update()
        - Broadcast order update via WebSocket
    - Call CheckOrderExpiry() in monitor loop (before trigger checks)

    Follow existing patterns: CORS, error handling, repository usage.
  </action>
  <verify>

    - curl -X DELETE localhost:8080/api/orders/1 returns 200
    - Manual test: Set expiry_time in past → order cancels on next monitor tick
  </verify>
  <done>
    - API allows modifying pending order parameters
    - API allows cancelling pending orders
    - Orders expire automatically after expiry_time
    - All changes persist to database
  </done>
