Read current lpmanager/manager.go implementation and refactor:

**Current pattern (if using slice/array)**:
```go
type Manager struct {
    lps []LP  // O(n) iteration to find LP
}

func (m *Manager) GetLP(name string) *LP {
    for i := range m.lps {
        if m.lps[i].Name == name {
            return &m.lps[i]
        }
    }
    return nil
}
```

**Refactored pattern (map-based)**:
```go
type Manager struct {
    lps map[string]*LP  // O(1) direct access by name/ID
    mu  sync.RWMutex    // Thread-safe access for concurrent lookups
}

func (m *Manager) GetLP(name string) *LP {
    m.mu.RLock()
    defer m.mu.RUnlock()
    return m.lps[name]  // O(1) lookup
}

func (m *Manager) AddLP(lp *LP) {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.lps[lp.Name] = lp
}

func (m *Manager) RemoveLP(name string) {
    m.mu.Lock()
    defer m.mu.Unlock()
    delete(m.lps, name)
}
```

**Key changes**:
- Replace []LP with map[string]*LP
- Add sync.RWMutex for thread safety (trading platform is concurrent)
- Initialize map in constructor: lps: make(map[string]*LP)
- Update all iteration logic to use map operations

Why map: O(1) lookup vs O(n) iteration. For 10+ LPs accessed thousands of times per second, this is significant.
