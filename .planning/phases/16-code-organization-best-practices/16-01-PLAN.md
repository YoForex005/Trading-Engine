# Plan: Linting Setup and Configuration

**Phase:** 16 - Code Organization & Best Practices
**Wave:** 1 (parallel with 16-02)
**Dependencies:** None
**Scope:** ~45% context budget

---

## Goal

Establish professional linting infrastructure with golangci-lint for Go backend and typescript-eslint for React frontend, catching bugs and enforcing consistent code style automatically.

---

## Must-Haves

### Truths (what must be TRUE when done)

1. golangci-lint runs successfully on backend codebase with `recommended` preset
2. typescript-eslint runs successfully on frontend codebase with `recommended` preset
3. CI/CD pipeline fails builds on linting violations
4. Both linters integrated into pre-commit workflow
5. No existing violations remain for critical rules (type errors, unused vars, etc.)

### Artifacts (files that must exist)

```
backend/.golangci.yml                    # golangci-lint configuration
clients/desktop/eslint.config.mjs        # Modern ESLint flat config
.github/workflows/lint.yml               # CI/CD linting workflow
```

### Key Links (code that must connect correctly)

- CI/CD workflow triggers on every commit (frontend + backend)
- Linters use `--new-from-rev=HEAD~1` in CI to only check changed code (faster)
- Exit codes propagate to fail builds on violations

## Tasks

### Task 1: Configure golangci-lint for Backend

**What:** Create `.golangci.yml` with recommended preset and project-specific rules

**Why:** Industry standard Go linter with 50+ integrated linters catches bugs early

**Details:**
```yaml
# .golangci.yml
linters:
  enable:
    - errcheck        # Check error returns
    - gosimple        # Simplify code
    - govet           # Go vet
    - ineffassign     # Detect ineffectual assignments
    - staticcheck     # Go static analysis
    - unused          # Find unused code
    - typecheck       # Type-check Go code
    - gofmt           # Check formatting
    - goimports       # Check imports
    - misspell        # Spell check
    - revive          # Drop-in replacement for golint

linters-settings:
  errcheck:
    check-blank: true  # Catch _ = err

  revive:
    rules:
      - name: exported
        disabled: false

run:
  timeout: 5m
  tests: true
  skip-dirs:
    - vendor
    - testdata

issues:
  exclude-use-default: false
  max-issues-per-linter: 0
  max-same-issues: 0
```

**Run:** `cd backend && golangci-lint run`

**Checkpoint:** Linter runs and shows issues (don't fix all yet - critical ones only in this plan)

---

### Task 2: Configure typescript-eslint for Frontend

**What:** Create `eslint.config.mjs` with typescript-eslint recommended preset

**Why:** Official TypeScript linting with type-aware rules catches type errors and enforces best practices

**Details:**
```javascript
// clients/desktop/eslint.config.mjs
import eslint from '@eslint/js';
import tseslint from 'typescript-eslint';

export default tseslint.config(
  eslint.configs.recommended,
  ...tseslint.configs.recommended,
  ...tseslint.configs.stylistic,
  {
    rules: {
      // Allow unused vars starting with _
      '@typescript-eslint/no-unused-vars': ['error', {
        argsIgnorePattern: '^_',
        varsIgnorePattern: '^_'
      }],

      // Warn on console.log (should use proper logging)
      'no-console': ['warn', { allow: ['warn', 'error'] }],

      // Enforce type imports
      '@typescript-eslint/consistent-type-imports': 'error',
    },
  },
);
```

**Install:** `cd clients/desktop && bun add -D @typescript-eslint/eslint-plugin @typescript-eslint/parser`

**Run:** `cd clients/desktop && bun run lint`

**Checkpoint:** Linter runs and shows issues

### Task 3: Fix Critical Violations Only

**What:** Fix high-priority issues that block CI/CD (type errors, unused imports, ineffectual assignments)

**Why:** Can't enable linting in CI/CD if critical issues fail the build

**Details:**
- Focus on TypeScript type errors (misconfigured types, missing imports)
- Fix unused imports that break builds
- Fix ineffectual assignments in Go (assigned but never used)
- **Do NOT refactor everything** - just make linters pass with minimal changes

**Examples:**
```go
// Before (ineffectual assignment)
result := calculateMargin(acc)  // never used
return acc

// After
_ = calculateMargin(acc)  // explicitly ignore
return acc

// Or remove if truly unused
```

```typescript
// Before (unused import)
import { Position, Order, Account } from './types';  // Account unused

// After
import { Position, Order } from './types';
```

**Checkpoint:** `golangci-lint run` and `bun run lint` exit with 0 (or acceptable warning count)

### Task 4: Add Linting to CI/CD Pipeline

**What:** Create GitHub Actions workflow to run linters on every commit

**Why:** Automate code quality checks, prevent regressions

**Details:**
```yaml
# .github/workflows/lint.yml
name: Lint

on:
  push:
    branches: [main]
  pull_request:

jobs:
  lint-go:
    name: Lint Go Code
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for --new-from-rev

      - uses: actions/setup-go@v5
        with:
          go-version: '1.22'

      - name: golangci-lint
        uses: golangci/golangci-lint-action@v4
        with:
          version: latest
          working-directory: backend
          args: --new-from-rev=HEAD~1  # Only lint changes

  lint-typescript:
    name: Lint TypeScript Code
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: oven-sh/setup-bun@v1

      - name: Install dependencies
        working-directory: clients/desktop
        run: bun install

      - name: Run ESLint
        working-directory: clients/desktop
        run: bun run lint
```

**Checkpoint:** Push to GitHub, workflow runs and passes

   Both should exit with code 0 (or acceptable warnings)

2. **Check CI/CD:**

   - Verify GitHub Actions workflow fails
   - Fix violation and verify workflow passes

3. **Verify configurations exist:**
   ```bash

   test -f clients/desktop/eslint.config.mjs && echo "✓ ESLint configured"
   test -f .github/workflows/lint.yml && echo "✓ CI/CD configured"
   ```

## Anti-Patterns to Avoid

- **Over-configuring linters:** Start with `recommended` presets, only add custom rules when needed
- **Fixing all violations in one PR:** Fix critical ones first, create follow-up tasks for warnings
- **Disabling rules without justification:** Document why a rule is disabled (e.g., `// nolint:errcheck // reason`)
- **Not running locally before commit:** Always run linters before pushing
