# Plan: Error Wrapping Standardization

**Phase:** 16 - Code Organization & Best Practices
**Wave:** 2 (after Wave 1 completes)
**Dependencies:** 16-01 (linting), 16-02 (structured logging)
**Scope:** ~40% context budget

---

## Goal

Standardize error handling across backend using Go error wrapping with fmt.Errorf("%w") and errors.Is/errors.As, adding contextual information to every error for production debugging.

---

## Must-Haves

### Truths (what must be TRUE when done)

1. All error returns include context via fmt.Errorf("context: %w", err)
2. No raw `return nil, err` or `return err` without wrapping
3. Error checking uses errors.Is/errors.As instead of == comparisons
4. Custom error types exported where needed for caller inspection
5. golangci-lint errcheck rule enabled and passing

### Artifacts (files that must exist)

```
backend/internal/shared/errors/errors.go  # Common error types
backend/.golangci.yml                      # errcheck enabled (updated from 16-01)
```

### Key Links (code that must connect correctly)

- Repository errors wrapped with entity context (account ID, position ID, etc.)
- HTTP handlers check errors with errors.Is for proper status codes
- Custom errors (NotFoundError, ValidationError) usable with errors.As

## Tasks

### Task 1: Create Common Error Types

**What:** Define reusable error types in `internal/shared/errors/`

**Why:** Standard error types for common scenarios (not found, validation, etc.)

**Details:**
```go
// backend/internal/shared/errors/errors.go
package errors

import "fmt"

// NotFoundError indicates a resource was not found
type NotFoundError struct {
	Resource string  // "account", "position", "order"
	ID       string
}

func (e *NotFoundError) Error() string {
	return fmt.Sprintf("%s not found: %s", e.Resource, e.ID)
}

func NewNotFound(resource, id string) error {
	return &NotFoundError{Resource: resource, ID: id}
}

// ValidationError indicates invalid input
type ValidationError struct {
	Field   string
	Message string
}

func (e *ValidationError) Error() string {
	return fmt.Sprintf("validation failed for %s: %s", e.Field, e.Message)
}

func NewValidation(field, message string) error {
	return &ValidationError{Field: field, Message: message}
}

// InsufficientFundsError indicates account lacks balance
type InsufficientFundsError struct {
	AccountID string
	Required  string  // decimal as string
	Available string
}

func (e *InsufficientFundsError) Error() string {
	return fmt.Sprintf("insufficient funds in account %s: required %s, available %s",
		e.AccountID, e.Required, e.Available)
}
```

**Checkpoint:** Package compiles, errors satisfy error interface

### Task 2: Wrap Repository Errors

**What:** Add context to all database repository errors

**Why:** Database errors lack business context (which account? which position?)

**Details:**

**Pattern for repositories:**
```go
// backend/internal/database/repository/account.go

import (
	"fmt"
	"errors as stderrors"

	"github.com/jackc/pgx/v5"
	"github.com/trading-engine/internal/shared/errors"
)

func (r *AccountRepository) Get(ctx context.Context, id string) (*Account, error) {
	var acc Account

	err := r.db.QueryRow(ctx, "SELECT * FROM accounts WHERE id = $1", id).Scan(&acc)
	if err != nil {
		// Check for specific error types
		if stderrors.Is(err, pgx.ErrNoRows) {
			// Return custom error type
			return nil, errors.NewNotFound("account", id)
		}
		// Wrap generic errors with context
		return nil, fmt.Errorf("failed to get account %s: %w", id, err)
	}

	return &acc, nil
}

func (r *AccountRepository) Update(ctx context.Context, acc *Account) error {
	_, err := r.db.Exec(ctx,
		"UPDATE accounts SET balance = $1, equity = $2 WHERE id = $3",
		acc.Balance, acc.Equity, acc.ID,
	)
	if err != nil {
		return fmt.Errorf("failed to update account %s: %w", acc.ID, err)
	}
	return nil
}
```

**Apply to all repositories:**
- `account.go` - Wrap with account_id context
- `position.go` - Wrap with position_id context
- `order.go` - Wrap with order_id context
- `trade.go` - Wrap with trade_id context
- `margin_state.go`, `risk_limit.go`, etc. - Wrap with relevant IDs

**Checkpoint:** All repository methods wrap errors with context

### Task 3: Wrap Business Logic Errors (bbook/engine.go)

**What:** Add context to errors in core trading engine

**Why:** engine.go orchestrates trading operations, needs clear error paths

**Details:**

Current patterns to fix:
```go
// BEFORE (raw error return, no context)
func (e *Engine) ExecuteBuy(order Order) error {
	if err := e.validateOrder(order); err != nil {
		return err  // ❌ No context
	}

	pos, err := e.createPosition(order)
	if err != nil {
		return err  // ❌ No context
	}

	return nil
}
```

After:
```go
// AFTER (wrapped with context)
func (e *Engine) ExecuteBuy(order Order) error {
	if err := e.validateOrder(order); err != nil {
		return fmt.Errorf("order validation failed for %s: %w", order.Symbol, err)
	}

	pos, err := e.createPosition(order)
	if err != nil {
		return fmt.Errorf("failed to create position for order %s: %w", order.ID, err)
	}

	// Log successful execution
	logger.Default.Info("buy order executed",
		"order_id", order.ID,
		"position_id", pos.ID,
		"symbol", order.Symbol,
	)

	return nil
}
```

**Focus on key functions:**
- `ExecuteBuy()` / `ExecuteSell()` - order execution errors
- `ClosePosition()` - position close errors
- `CalculateMargin()` - margin calculation errors
- `ExecuteStopOut()` - liquidation errors
- All functions returning error type

**Checkpoint:** engine.go has no raw `return err` without wrapping

### Task 4: Update HTTP Handlers Error Checking

**What:** Use errors.Is/errors.As in API handlers for proper status codes

**Why:** Different errors map to different HTTP status codes

**Details:**

**Pattern for handlers:**
```go
// backend/bbook/api.go

import (
	stderrors "errors"

	"github.com/trading-engine/internal/shared/errors"
)

func (h *Handler) GetAccount(c *gin.Context) {
	accountID := c.Param("id")

	acc, err := h.accountRepo.Get(c.Request.Context(), accountID)
	if err != nil {
		// Check error type for appropriate status code
		var notFoundErr *errors.NotFoundError
		if stderrors.As(err, &notFoundErr) {
			c.JSON(404, gin.H{"error": "account not found"})
			return
		}

		// Generic error
		logger.Default.Error("failed to get account",
			"account_id", accountID,
			"error", err,
		)
		c.JSON(500, gin.H{"error": "internal server error"})
		return
	}

	c.JSON(200, acc)
}

func (h *Handler) PlaceOrder(c *gin.Context) {
	var order Order
	if err := c.ShouldBindJSON(&order); err != nil {
		c.JSON(400, gin.H{"error": err.Error()})
		return
	}

	if err := h.engine.ExecuteBuy(order); err != nil {
		// Check for validation errors
		var valErr *errors.ValidationError
		if stderrors.As(err, &valErr) {
			c.JSON(400, gin.H{"error": valErr.Message, "field": valErr.Field})
			return
		}

		// Check for insufficient funds
		var fundsErr *errors.InsufficientFundsError
		if stderrors.As(err, &fundsErr) {
			c.JSON(422, gin.H{
				"error": "insufficient funds",
				"required": fundsErr.Required,
				"available": fundsErr.Available,
			})
			return
		}

		// Generic error
		logger.Default.Error("order execution failed", "error", err)
		c.JSON(500, gin.H{"error": "internal server error"})
		return
	}

	c.JSON(200, gin.H{"status": "success"})
}
```

**Apply pattern to all handlers:**
- GET endpoints → 404 for NotFoundError, 500 for others
- POST/PUT endpoints → 400 for ValidationError, 422 for business logic errors, 500 for system errors
- DELETE endpoints → 404 for NotFoundError, 409 for conflict (e.g., position still open)

**Checkpoint:** Handlers use errors.As for type checking, return appropriate status codes

### Task 5: Enable errcheck Linter

**What:** Update .golangci.yml to enforce error checking

**Why:** Catch unchecked errors automatically

**Details:**

```yaml
# backend/.golangci.yml (update from 16-01)
linters:
  enable:
    - errcheck        # Now strictly enforced

linters-settings:
  errcheck:
    check-blank: true           # Catch _ = err (intentional ignore)
    check-type-assertions: true # Catch unchecked type assertions
    exclude-functions:
      - fmt.Fprintf
      - (io.Closer).Close       # Allow defer file.Close() without check
```

**Run and fix violations:**
```bash
cd backend
golangci-lint run --enable errcheck
```

**Common fixes:**
```go
// BEFORE (unchecked error)
db.Close()

// AFTER (explicit check)
if err := db.Close(); err != nil {
	logger.Default.Warn("failed to close database", "error", err)
}

// OR (intentional ignore with comment)
_ = db.Close()  // Already handled in defer, ignore close error
```

**Checkpoint:** `golangci-lint run` passes with errcheck enabled

### Task 6: Add Error Handling Documentation

**What:** Document error handling patterns for team

**Why:** Ensure consistent error handling in future code

**Details:**

Add to CONTRIBUTING.md:
```markdown
## Error Handling

### Always Wrap Errors with Context

Use `fmt.Errorf` with `%w` to preserve error chain:

```go
if err := repo.Get(ctx, id); err != nil {
	return fmt.Errorf("failed to get account %s: %w", id, err)
}
```

### Use Custom Error Types for Business Logic

```go
import "github.com/trading-engine/internal/shared/errors"

// Return typed errors
if balance < required {
	return errors.NewInsufficientFunds(accountID, required, balance)
}

// Check error types in callers
var fundsErr *errors.InsufficientFundsError
if errors.As(err, &fundsErr) {
	// Handle insufficient funds specifically
}
```

### Check Errors with errors.Is/errors.As

```go
// Check error identity
if errors.Is(err, pgx.ErrNoRows) {
	return errors.NewNotFound("position", id)
}

// Extract error type
var valErr *ValidationError
if errors.As(err, &valErr) {
	return 400, valErr.Message
}
```

### Never Ignore Errors

```go
// BAD
repo.Update(ctx, account)  // Error ignored

// GOOD
if err := repo.Update(ctx, account); err != nil {
	return fmt.Errorf("failed to update account: %w", err)
}

// ACCEPTABLE (with comment)
_ = file.Close()  // Defer handles error, ignore here
```
```

**Checkpoint:** Documentation is clear and includes examples

## Anti-Patterns to Avoid

- **Over-wrapping:** Don't wrap errors 5+ times in the same call stack (wrap at boundaries)
- **Losing error type:** Don't use fmt.Errorf without %w when error type matters
- **Ignoring sentinel errors:** Check for sql.ErrNoRows, io.EOF, context.Canceled with errors.Is
- **Generic error messages:** Add specific context (IDs, values) to every wrap
